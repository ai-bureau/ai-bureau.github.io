# Правила работы AI-агентов

Этот файл обязателен для всех AI-агентов, работающих с репозиторием, Notion или
GitHub проекта AI Bureau.

## Главное правило: сначала понять, потом действовать

Перед кодом, изменением Notion, публикацией, планом или документом:

1. Изучи задачу и существующий контекст.
2. Самостоятельно прочитай необходимые файлы и проверь состояние систем.
3. Если информации недостаточно и её нельзя получить самостоятельно — задай
   вопрос до начала изменений.
4. Коротко изложи план.
5. Дождись явного подтверждения пользователя: `давай`, `go`, `вперёд` или
   аналогичного сигнала.

Не начинай значимую реализацию на основании первого сообщения.

## Цель проекта

AI Bureau передаёт проверяемую рутину машине, сохраняя решения и
ответственность за человеком. Система контента должна работать максимально
автоматически, но публикация всегда требует явного статуса `Публікувати`.

## Источники правды

В порядке приоритета:

1. Фактическое состояние кода, GitHub и Notion.
2. Документы в `docs/` и актуальный `README.md`.
3. Страница проекта в Notion.
4. Старые ТЗ `tz-*.md`.

Если источники расходятся, не делай скрытых предположений. Зафиксируй
расхождение и используй фактическую систему, если пользователь не указал иное.

## Границы автономности

Разрешено без дополнительного вопроса после общего подтверждения задачи:

- читать файлы, схемы Notion, логи и состояние GitHub;
- запускать тесты, Hugo-сборку и dry-run;
- исправлять подтверждённую задачу в рамках существующей архитектуры;
- обновлять документацию вслед за кодом;
- искать статьи по утверждённым правилам.

Требует отдельного явного подтверждения:

- публикация статьи без статуса `Публікувати`;
- удаление страниц или баз Notion;
- изменение схемы Notion;
- удаление опубликованной статьи;
- ротация или отзыв токенов;
- смена домена, GitHub Pages source или основной ветки;
- разрушительные Git-команды;
- изменение позиционирования AI Bureau.

## Работа с контентом

- Поиск статей выполняется вручную агентом, не автоматическим Article Finder.
- Допустимые языки публикации: `UA` и `EN`.
- Русскоязычные материалы не добавляются.
- Статья должна подтверждать конкретный тезис, а не просто касаться темы.
- Проверяй оригинальный источник, автора, дату и позицию материала.
- Проверяй дубликаты URL в базе `Статьи`.
- Добавляй честное саммари: что утверждает материал и как он связан с тезисом.
- Перед принятием тезиса учитывай сильный контраргумент, чтобы не строить
  некритичную машину подтверждения собственных убеждений.
- Не меняй статус на `Публікувати` за пользователя, если он явно не попросил.

Подробности: `docs/CONTENT_WORKFLOW.md`.

## Работа с публикатором

- Сначала запускай:

  ```bash
  python -m publisher --env-file actual.env --dry-run
  ```

- Реальный запуск выполняй только после проверки dry-run и явного разрешения.
- Не выводи содержимое `actual.env` и токены в ответах, логах или командах.
- Не обновляй Notion вручную до `Опубліковано`, если GitHub-коммит не создан.
- При частичном сбое повтори запуск: публикатор идемпотентен для идентичного
  файла.
- После изменений публикатора запускай все тесты.

## Работа с GitHub и сайтом

- Репозиторий: `ai-bureau/ai-bureau.github.io`.
- Основная ветка: `main`.
- Деплой: `.github/workflows/hugo.yaml`.
- Не коммить `public/`, `.tools/`, `logs/`, `.env`, `actual.env`.
- PaperMod подключён submodule и должен оставаться закреплённым на конкретном
  commit.
- Перед push обязательно:

  ```bash
  python -m pytest
  .tools/hugo/hugo --gc --minify --destination /tmp/ai-bureau-check
  git diff --check
  ```

- После push проверь Actions и публичные UA/EN маршруты.

## Качество кода

- Каждая функция имеет одну ответственность и docstring.
- Используй описательные имена.
- Диагностика только через стандартный `logging`.
- Не глотай исключения.
- Внешние API и файлы обрабатывай с явными ошибками и контекстом.
- Проверяй входные данные на границах.
- Не добавляй абстракции без реальной необходимости.
- Новое поведение должно иметь тесты.
- После значимого изменения сразу обновляй документацию.

- какие проверки прошли;
- какие нестандартные решения приняты и почему;
- что осталось за рамками задачи.
