# Архитектурные решения

## ADR-001: Hugo и GitHub Pages

**Решение:** статический сайт собирается Hugo и публикуется GitHub Actions.

**Почему:** минимальная стоимость, высокая скорость, простое версионирование и
отсутствие постоянно работающего backend.

## ADR-002: PaperMod как основа, собственный дизайн поверх

**Решение:** использовать PaperMod для инфраструктуры блога и i18n, но
переопределить главную, статью и стили.

**Почему:** готовые SEO/i18n-механизмы без стандартного вида технического блога.

## ADR-003: Notion как контентный интерфейс

**Решение:** Notion раздельно хранит тезисы, источники, готовые двуязычные
публикации и решение о публикации.

**Почему:** удобен человеку и AI, достаточен для небольшого контентного
pipeline. Он не объявляется универсальной БД для всех будущих систем.

## ADR-004: Ручной поиск статей

**Решение:** Codex ищет и оценивает статьи вручную; автоматический Article
Finder не реализуется.

**Почему:** соответствие материала позиции требует содержательной оценки,
проверки источника и контроля confirmation bias.

## ADR-005: Человек явно разрешает публикацию

**Решение:** единственный триггер публикации — статус `Публікувати`.

**Почему:** сохраняет человека в контуре принятия решения и предотвращает
автономную публикацию неподтверждённого материала.

## ADR-006: Одноразовый публикатор

**Решение:** каждый запуск выполняет один проход и завершается.

**Почему:** публикации редкие; постоянный сервис, NSSM и частый polling
избыточны. Позже скрипт можно запускать любым планировщиком без изменения ядра.

## ADR-007: Прямые HTTP API вместо SDK

**Решение:** использовать `requests` для Notion и GitHub API.

**Почему:** небольшой набор операций, прозрачные контракты, меньше зависимостей,
явное использование актуального Notion Data Source API.

## ADR-008: GitHub изменяется раньше Notion

**Решение:** сначала создать GitHub-файл, затем отметить Notion опубликованным.

**Почему:** статус `Опубліковано` не должен появляться без фактического
публикационного артефакта. Идентичный файл позволяет восстановиться после
частичного сбоя.

## ADR-009: Отдельные каталоги языков

**Решение:** `content/uk/` и `content/en/`.

**Почему:** явная и понятная маршрутизация для Hugo и публикатора.

## ADR-010: Скрывать будущие статьи средствами Hugo

**Решение:** файл создаётся сразу, показ контролируется `publishDate`.

**Почему:** не требуется дополнительный scheduler для отложенной публикации.

## ADR-011: Одна карточка — одна двуязычная публикация

**Решение:** готовые UA и EN версии хранятся в одной странице `Публикации`.

**Почему:** обе версии проходят одно человеческое одобрение, имеют общий slug и
не могут разойтись по статусу.

## ADR-012: Публикатор не создаёт содержание

**Решение:** публикатор только валидирует и публикует готовый текст без
перевода, сокращения или дописывания.

**Почему:** содержание и авторская позиция должны быть явно проверены до
публикации.

## ADR-013: Атомарный коммит UA + EN

**Решение:** обе версии создаются одним GitHub-коммитом через Git Data API.

**Почему:** сайт не должен оказаться в состоянии, где опубликована только одна
из одобренных языковых версий.
